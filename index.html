Perjalanan Teori Normalisasi Struk Listrik

UNF

Bentuk Tidak Normal (Unnormalized Form)

Data mentah langsung ditulis apa adanya dari dokumen fisik

Pada kondisi ini, seluruh data dari struk (seperti Nama Konter, IDPEL, Nominal, hingga Token) dikumpulkan ke dalam satu tabel datar yang sangat besar tanpa struktur yang baik.

{ No_Transaksi, Nama_Konter, Waktu_Transaksi, IDPEL, Nama_Pelanggan, Tarif_Daya, Nominal, Rp_Token, Jml_KWh, Adm_Bank, Admin_Konter, Total_Bayar, Nomor_Token }
1NF

Bentuk Normal Pertama (1st Normal Form)

Nilai data harus atomik (tidak bercabang)

Pada struk asli, terdapat kolom TRF/DAYA : R1/450. Kolom ini menggabungkan dua informasi penting. Sesuai prinsip 1NF, kita pecah menjadi dua nilai atomik yaitu: **Golongan Tarif** (R1) and **Daya Listrik** (450).

[Tabel_1NF] = { No_Transaksi, Nama_Konter, Waktu_Transaksi, ID_Pelanggan, Nama_Pelanggan, Gol_Tarif, Daya, Nominal, Rp_Token, Jml_KWh, Adm_Bank, Admin_Konter, Total_Bayar, Nomor_Token }
2NF

Bentuk Normal Kedua (2nd Normal Form)

Menghilangkan Ketergantungan Parsial (Partial Dependency)

Atribut master pelanggan seperti Nama_Pelanggan, Gol_Tarif, Daya sebenarnya hanya bergantung pada ID_Pelanggan, bukan pada kunci transaksi No_Transaksi. Jika kita tidak memisahkan mereka, maka setiap pelanggan melakukan transaksi baru, kita harus menduplikasi nama dan daya mereka. Kita pecah menjadi tabel terpisah:

1. Tabel Pelanggan: { ID_Pelanggan (PK), Nama_Pelanggan, Gol_Tarif, Daya }
2. Tabel Transaksi_Token: { No_Transaksi (PK), ID_Pelanggan (FK), Nama_Konter, Waktu_Transaksi, Nominal, Rp_Token, Jml_KWh, Adm_Bank, Admin_Konter, Total_Bayar, Nomor_Token }
Standard Akhir
3NF

Bentuk Normal Ketiga (3rd Normal Form)

Menghilangkan Ketergantungan Transitif & Redundansi Sekunder

Ada dua masalah serius di 2NF yang diselesaikan di tahap 3NF ini demi mencapai integritas data:

  1. Masalah Nama Konter (Redundansi String): Nama konter GENTENG CELL1 tidak boleh ditulis berulang di ribuan baris transaksi. Kita pindahkan ke tabel Konter.
  2. Masalah Total Bayar (Ketergantungan Transitif): Total_Bayar didapat secara mutlak dari rumus matematik: Nominal + Adm_Bank + Admin_Konter. Kolom kalkulasi ini dipindahkan ke tabel detail biaya terpisah, atau dihitung di tingkat query/aplikasi untuk menghindari anomali perubahan biaya admin.

-- 1. Buat Tabel Master Pelanggan
CREATE TABLE Pelanggan (
    ID_Pelanggan VARCHAR(12) PRIMARY KEY,
    Nama_Pelanggan VARCHAR(100) NOT NULL,
    Gol_Tarif VARCHAR(5) NOT NULL,
    Daya INT NOT NULL
);

-- 2. Buat Tabel Master Konter
CREATE TABLE Konter (
    ID_Konter INT AUTO_INCREMENT PRIMARY KEY,
    Nama_Konter VARCHAR(50) NOT NULL UNIQUE
);

-- 3. Buat Tabel Utama Transaksi Token
CREATE TABLE Transaksi_Token (
    No_Transaksi VARCHAR(20) PRIMARY KEY,
    ID_Konter INT NOT NULL,
    ID_Pelanggan VARCHAR(12) NOT NULL,
    Waktu_Transaksi DATETIME NOT NULL,
    Rp_Token DECIMAL(10,2) NOT NULL,
    Jml_KWh DECIMAL(6,2) NOT NULL,
    Nomor_Token VARCHAR(20) NOT NULL UNIQUE,
    FOREIGN KEY (ID_Konter) REFERENCES Konter(ID_Konter) ON DELETE RESTRICT ON UPDATE CASCADE,
    FOREIGN KEY (ID_Pelanggan) REFERENCES Pelanggan(ID_Pelanggan) ON DELETE RESTRICT ON UPDATE CASCADE
);

-- 4. Buat Tabel Detail Biaya Transaksi (Pemisahan Total_Bayar demi 3NF)
CREATE TABLE Detail_Biaya_Transaksi (
    No_Transaksi VARCHAR(20) PRIMARY KEY,
    Nominal DECIMAL(10,2) NOT NULL,
    Adm_Bank DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    Admin_Konter DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    -- MariaDB / PostgreSQL / MySQL Generated Column:
    Total_Bayar DECIMAL(10,2) GENERATED ALWAYS AS (Nominal + Adm_Bank + Admin_Konter) STORED,
    FOREIGN KEY (No_Transaksi) REFERENCES Transaksi_Token(No_Transaksi) ON DELETE CASCADE
);
